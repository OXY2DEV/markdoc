üí• Features                                  *markdoc-features* *markdoc-feat*

  ‚Ä¢ Config per file, allows configuring `markdoc` using YAML metadata.
  ‚Ä¢ Tag(s) per heading, allows using Lua patterns to add one or more tags to
    matching headings.
  ‚Ä¢ Creating TOCs, allows creating Table of contents via YAML.
  ‚Ä¢ Callout & title support, supports callouts(even custom ones) and titles
    for block quotes.
  ‚Ä¢ Prettier table rendering, tables can be created with custom borders with
    text wrapping support.
  ‚Ä¢ Nesting support, supports nesting of block elements.
  ‚Ä¢ Base HTML support, supports basic HTML tags.
  ‚Ä¢ `align=""` support, allows aligning text via HTML.

üìö Usage                                                       *markdoc-usage*

You only need the `markdoc.lua` file. So, just copy it to your desired
location.

You can then run it like so,

>shell
  pandoc -t path/to/markdoc.lua README.md -o help.txt
<

‚ñå üí° Tip
‚ñå You can store metadata in a separate file too if you like. Then you will do
‚ñå something like this,

  >sh
    # Load the metadata file first
    pandoc metadata.md README.md -t path/to/markdoc.lua -o help.txt
<

üí® Github actions                                            *markdoc-actions*

‚ñå üö® Warning
‚ñå I am not very knowledgable when it comes to GitHub actions, so feel free to
‚ñå do PR(s) to fix issues I missed.

>yml
  name: markdoc
  on: [ push ]
  
  permissions:
    contents: write
  
  jobs:
    doc:
      runs-on: ubuntu-latest
      name: "To vimdoc"
      steps:
        - uses: actions/checkout@v2
        - uses: OXY2DEV/markdoc@v2
          with:
            config: '{ "doc/markdoc.txt": [ "mREADME.md", "README.md" ] }'
        - uses: stefanzweifel/git-auto-commit-action@v4
          with:
            commit_message: "doc: Generated help files"
            branch: ${{ github.head_ref }}
<

This basically runs `git checkout`, pandoc mREADME.md README.md -t markdoc.lua
-o doc/markdoc.txt, `git add .` and git commit -m "doc: Generated help files".

üìå Configuring workflow ~

The workflow comes with a single option, `config`. It‚Äôs a JSON string that
actually looks like this,

>json
  {
      "doc/markdoc.txt": [ "mREADME.md", "README.md" ]
  }
<

This generates `markdoc.txt` from `mREADME.md`(this file is used for metadata)
and `README.md`.

You can add more keys to create more files. For example,

>json
  {
      "doc/markdoc.txt": [ "mREADME.md", "README.md" ],
      "doc/markdoc-arch.txt": [ "ARCHITECTURE.md" ]
  }
<

This will generate another help file named `markdoc-arch` from the
`ARCHITECTURE.md` file.

------------------------------------------------------------------------------

As of `v3` there is also a `help_dirs` option. This is a JSON list of
directories where `helptags` will be run.

>json
  [ "doc" ]
<

By default it only runs on the `doc/` directory.

ü™® Limitations                           *markdoc-limitations* *markdoc-limit*

‚ñå üß© Important
‚ñå `markdoc` is best-effort only. It‚Äôs not perfect but for most use cases it
‚ñå should be good enough.

  ‚Ä¢ Whitespace loss, due to the nature of syntax trees the whitespace in the
    document may not be preserved.
  ‚Ä¢ Issues with complex characters, some UTF-8 characters don‚Äôt get properly
    wrapped(either due to multi-byte width or limitation of Lua).
  ‚Ä¢ Issues with complex document structures, though `markdoc` should support
    complex documents it is still preferable to keep them simple.
    
    You may encounter text wrapping issues if the document gets too deeply
    nested(due to lack of space).

üî© Configuration                                       *markdoc-configuration*
                                                              *markdoc-config*

                        ‚ÄúOne function to rule them all,                       

                          One function to find them,                          

                        One function to bring them all,                       

                        And in the darkness bind them.‚Äù                       

You can configure `markdoc` for a file by adding a YAML metadata section to
the start of the file.

It should look something like this,

>txt
  ---
  markdoc:
    textwidth: 78
    title: "Markdown ü§ù Vimdoc"
    title_tag: "markdoc"
  
    tags:
      Features$: [ "markdoc-features", "markdoc-feat" ]
    Usage$: [ "markdoc-usage" ]
  ---
  Some text.
<

‚ñå üìú Note
‚ñå All configuration options should be inside `markdoc`.

The configuration options are given below,

block_quotes ~

Changes block quote configuration.

EXAMPLE

>yaml
    block_quotes:
      default:
        border: "‚ïë"
  
      # Name SHOULD be in lowercase.
      # This will match any variation of,
      # [!CUSTOM]
      custom:
        border: "‚îä" # Block quote border
        callout: "üéµ Custom" # Text to show at the top
        icon: "üéµ" # Icon before title(if there is any).
<

fold_refs ~

When `true`, link & image references are folded.

EXAMPLE

>yaml
    fold_refs: true
<

foldmarkers ~

Markers used for folding.

EXAMPLE

>yaml
    # This is the same as `:set foldmarker`
    foldmarkers: "{{{,}}}"
<

tags ~

Tag configuration for different headings.

EXAMPLE

>yaml
      tags:
        # The key is a Lua patter.
        Features$: [ "markdoc-features", "markdoc-feat" ] # Adds 2 tags to headings ending with Feature.
        Usage$: [ "markdoc-usage" ]
<

table ~

Table configuration.

EXAMPLE

>yaml
      table:
        col_minwidth: 10 # Minimum width for columns.
  
        top: [ "‚ï≠", "‚îÄ", "‚ïÆ", "‚î¨" ]
        header: [ "‚îÇ", "‚îÇ", "‚îÇ" ]
  
        separator: [ "‚îú", "‚îÄ", "‚î§", "‚îº" ]
        header_separator: [ "‚îú", "‚îÄ", "‚î§", "‚îº" ]
        row_separator: [ "‚îú", "‚îÄ", "‚î§", "‚îº" ]
  
        row: [ "‚îÇ", "‚îÇ", "‚îÇ" ]
        bottom: [ "‚ï∞", "‚îÄ", "‚ïØ", "‚î¥" ]
<

textwidth ~

Width of the document.

EXAMPLE

>yaml
    # Same as `:set textwidth`
    textwidth: 78
<

title ~

Title of the document.

EXAMPLE

>yaml
    title: "Markdown ü§ù Vimdoc"
<

title_tag ~

Tag for the document title.

EXAMPLE

>yaml
    title_tag: "markdoc"
<

toc_title ~

Title for the generated table of contents section.
Defaults to `Table of contents:` when unset.

EXAMPLE

>yaml
    toc_title: "Markdoc: TOC"
<

toc ~

Entries for the table of contents.

‚ñå üí° Tip

  You can add `<MKDocTOC/>` to anywhere in your document and the table of
  contents will be placed there!
  However, It will also only replace the 1st one.

EXAMPLE

>yaml
    toc:
      "Features": "markdoc-features"
<
